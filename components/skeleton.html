Skeleton

Skeleton screens are placeholder UI shown while content is loading. They reduce perceived loading time by showing the approximate shape of content before it arrives, preventing layout shift.

When to Use

Use Skeleton when
  • Initial page or section load (data fetch on mount)
  • Table rows while fetching records
  • Card content before data arrives
  • Profile information while loading user
Don't use Skeleton when
  • Button loading states — use .btn-loading spinner
  • Operations under 300ms — skip skeleton, show content directly
  • Form submission — use button loading state

Anatomy

1
Base class
div.skeleton. Background gradient animation (shimmer). background-size:200% 100% sweeping left to right. Duration: 1.5s infinite.
2
Shape modifiers
.skeleton-circle (border-radius:50%), .skeleton-text (height:12px), .skeleton-heading (height:18px), .skeleton-block (custom height). Width set inline or via container.
3
Layout wrapper
The skeleton mimics the exact layout of the content it replaces — same grid, same gaps, same proportions. This prevents layout shift when real content loads.

Variants

States

Skeleton has one state — loading. It renders while data is being fetched and unmounts completely when data arrives. There is no visible transition from skeleton to content — the swap is instant. The shimmer animation runs continuously at 1.5s and stops only when the skeleton unmounts.

Behavior

TriggerBehavior
Data loadingRender skeleton elements matching content layout. Shimmer animation runs continuously at 1.5s.
Data arrivesUnmount skeleton, mount real content. No animation — instant swap. Real content appears at exact same position as skeleton.
Error stateUnmount skeleton, render error empty state instead.

Accessibility

RequirementImplementation
Screen readerAdd aria-busy="true" to the container while skeleton is showing. Remove when content loads.
Reduce motionRespect @media (prefers-reduced-motion): disable shimmer animation. Show static gray blocks instead.

Best Practices

Do
Mirror the exact layout of real content — same grid, same gaps, same proportions.
Prevents layout shift when content loads. Builds accurate user expectation.
Don't
Show a generic spinning loader instead of skeleton for section-level loading.
Spinner gives no hint about the shape of incoming content, causing disorienting layout shifts.

Content Rules

Skeleton elements have no text content. Their size and shape communicate the expected content layout. Use realistic proportions — a heading skeleton should be ~50-70% of container width, body text 75-100%, sub-text 30-50%.

AI Implementation Guide

For AI coding assistants: CSS: background: linear-gradient(90deg, #F1F5F9 25%, #E2E8F0 50%, #F1F5F9 75%); background-size:200% 100%; animation:shimmer 1.5s infinite. Skeleton elements match content layout exactly — same DOM structure, same widths. When data loads, swap skeleton tree for real content tree. Add aria-busy="true" to container while loading. For table loading, show 3-5 skeleton rows. For KPI grid, show 4 skeleton cards.

Design Decisions

Why sweep from left to right (not pulse)?
The directional sweep mimics natural reading direction — it implies content is "arriving" from the source (left). A simple pulse (opacity fade) doesn't communicate directionality and feels more like a generic "waiting" indicator. The sweep creates a stronger perception of progress.
